## Context

現在のアプリは起動時に `100 × 10枚` の単一チップ構成で初期化される。ポーカープレイヤーはゲーム種別（キャッシュゲーム/トーナメント）やブラインドレベルに応じて異なるチップ構成を使うため、毎回手動で設定し直す必要がある。

既存のストレージ構成:
- `chips` → sessionStorage（タブ閉じで消える）
- `stack-session` → localStorage（セッション間で永続）

既存の shadcn/ui コンポーネント（Dialog, Button, ScrollArea）で UI は実装可能。

## Goals / Non-Goals

**Goals:**
- ポーカーで一般的なチップ構成をビルトインプリセットとして提供し、ワンタップでチップ+ブラインドを一括セットできる
- ユーザーが現在のチップ構成をカスタムプリセットとして保存・復元できる
- プリセット機能が既存機能（チップ編集、スタック記録）と干渉しない

**Non-Goals:**
- プリセットの import/export（JSON ファイル等）
- プリセットの並び替え・フォルダ分類
- ブラインドスケジュール（トーナメントのレベル進行）の管理
- プリセットの共有機能

## Decisions

### 1. プリセットデータ構造

```typescript
interface PresetChip {
  amount: number
  unit: Unit
  count: number
  color: string
}

interface ChipPreset {
  id: string
  name: string
  category: "system" | "user"
  blindAmount: number
  blindUnit: Unit
  chips: PresetChip[]
}
```

ChipRow の `id` はプリセットに含めない。読み込み時に動的に採番する。`count` はプリセットに含めることで「典型的な 100BB バイインスタック」をそのまま復元できる。

**代替案**: `count` を含めず額面のみ保存 → ゲーム中に途中保存した枚数ごと復元したいユースケースに対応できないため却下。

### 2. ビルトインプリセット定義

ポーカーの一般的なチップカラー規約に基づく:

| 色 | Hex | 代表的な額面 |
|---|---|---|
| White | `#e5e7eb` | $1, T25 |
| Red | `#ef4444` | $5, T100 |
| Green | `#22c55e` | $25, T500 |
| Black | `#1f2937` | $100, T1000 |
| Purple | `#a855f7` | $500, T5000 |

**キャッシュゲーム（100BB バイイン想定）:**

| プリセット名 | BB | チップ構成 | 合計 |
|---|---|---|---|
| Cash 1/2 | $2 | $1×10, $5×18, $25×4 | $200 |
| Cash 1/3 | $3 | $1×15, $5×17, $25×8 | $300 |
| Cash 2/5 | $5 | $5×20, $25×8, $100×2 | $500 |
| Cash 5/10 | $10 | $5×10, $25×18, $100×5 | $1,000 |

**トーナメント:**

| プリセット名 | BB | チップ構成 | 合計 |
|---|---|---|---|
| Tournament 5K | T50 | T25×8, T100×8, T500×4, T1000×2 | T5,000 |
| Tournament 30K | T200 | T100×8, T500×2, T1000×8, T5000×4 | T30,000 |

### 3. ストレージ設計

ユーザーカスタムプリセットは localStorage の `chip-presets` キーに配列として保存する。ビルトインプリセットはコード内定数として持ち、ストレージには保存しない。

```
localStorage:
  "chip-presets" → ChipPreset[] (ユーザー保存分のみ)
  "stack-session" → Session (既存)
sessionStorage:
  "chips" → ChipRow[] (既存)
  "current-blind-*" → (既存)
```

カテゴリは `"system"`（ビルトイン）と `"user"`（ユーザー保存）の 2 種のみ。ゲーム種別（cash/tournament）の区別はプリセット名で表現すれば十分。

**代替案**: ビルトインもストレージに入れてユーザーが編集可能にする → 複雑性が増す割にメリットが薄いため却下。ビルトインは読み取り専用とする。

### 4. UI 配置

Chips セクションヘッダー行に「Presets」ボタンを配置し、クリックでプリセット一覧ダイアログを表示する。各プリセット行にはチップアイコン（色と額面）のプレビューを表示し、どんなチップ構成か一目で分かるようにする。system プリセットと user プリセットをセクション分けして一覧表示する。

**代替案 A**: ドロップダウンセレクト → プリセット数が増えると使いにくく、保存/削除のアクションが入れにくい。
**代替案 B**: Sheet（ボトムシート）→ モバイルでは良いがデスクトップで違和感。Dialog なら既存パターンと統一。

### 5. プリセット読み込み時の動作

プリセット読み込み時は、現在のチップリスト全体とブラインド値を**上書き**する。確認ダイアログは出さない（undo 操作はないが、チップは即座に手動編集可能なため影響が軽い）。

## Risks / Trade-offs

- **[ビルトインプリセットの網羅性]** → すべてのゲーム形式をカバーすることは不可能。主要な 6 パターンに絞り、カスタムプリセットで補完する方針。
- **[プリセット読み込みで現在の枚数が消える]** → 確認ダイアログなしの上書きとするが、sessionStorage に直前の状態が残るため、リロードで復元可能（ただしこれは意図的な動作ではなく副次的な安全網）。
- **[localStorage 容量]** → プリセット1件あたり数百バイト程度。100件保存しても数十KBで問題なし。
