## Context

現在の `ChipEditForm.tsx` は `chipColors` 配列（10色固定）をグリッド表示しており、色選択は `onClick` で hex 文字列を親に渡すだけのシンプルな設計。`ChipRow.color` は `string` 型で任意の hex 値を格納でき、`getTextColor()` も任意の hex に対応済み。つまり、データレイヤーは既に拡張カラーに対応しているため、変更はUIレイヤーのみで完結する。

## Goals / Non-Goals

**Goals:**
- 9色プリセットパレット＋「+」ボタンの5×2グリッドで、拡張カラーピッカーへ自然に誘導する
- 任意の hex カラーコードを直接入力できるようにする
- 選択中の色をプレビュー表示する
- 既存UIの操作感を損なわない（クイック選択は引き続き使える）

**Non-Goals:**
- HSL スライダーやカラーホイールのような自由度の高いカラーピッカー（複雑すぎる）
- アルファチャネル（透明度）のサポート
- カスタムカラーのユーザーパレット保存機能（将来の拡張候補）
- `<input type="color">` ブラウザネイティブカラーピッカーの利用（モバイルでの UX が不安定）

## Decisions

### 1. HSLベースのカラーグリッド生成

**決定**: Hue を 16段階（22.5°刻み）、Saturation を 2段階（70%, 100%）、Lightness を 8段階（15%〜90%）で組み合わせたグリッドを生成する（16 × 16 = 256色）。

**理由**: RGB の均等分割は人間の色知覚と合わない。HSL ベースなら「赤系」「青系」のように色相で直感的にグループ化できる。Hue をX軸、Lightness をY軸にすれば、縦方向で明暗を調整する自然なUIになる。

**代替案**:
- RGB 6×6×6 キューブ → 216色だが中間色が偏る
- CSS Named Colors → 色名は覚えやすいが数が不均一
- ブラウザネイティブ `<input type="color">` → モバイルで挙動が不安定、カスタマイズ不可

### 2. 「+」ボタンによるインライン拡張

**決定**: プリセットを Gray を除いた9色にし、5×2グリッドの10番目のスロットに「+」ボタンを配置する。クリックでグリッド下に拡張カラーピッカーを展開する。

**理由**: グリッドの空きスロットを有効活用し、独立したボタンを追加せずに自然な導線を作れる。「+」は「追加の色を選ぶ」という意図が直感的に伝わる。Gray はポーカーチップとして使用頻度が低いため除外対象として適切。

**代替案**:
- パレット下にテキストボタン「もっと色を見る」→ 動作は同じだがグリッドの統一感が損なわれる
- 別ダイアログで開く → 画面遷移が増えて煩雑
- 最初から全色表示 → ダイアログが大きくなりすぎ、選択に迷う

### 3. Hex 直接入力

**決定**: 拡張ピッカー展開時にテキスト入力フィールドを表示し、`#` + 6桁の hex をバリデーション付きで入力可能にする。

**理由**: 特定の色コードを指定したいパワーユーザーのニーズに対応。入力値は `#` プレフィックスを自動付与し、不正値は適用しない。

### 4. コンポーネント分割

**決定**: 拡張カラーピッカーを `ExtendedColorPicker.tsx` として独立コンポーネントにする。`ChipEditForm.tsx` からは props で `color` と `onColorChange` を渡す。

**理由**: 既存の `ChipEditForm` の責務を増やさず、カラーピッカーのロジック（HSLグリッド生成、hex バリデーション）を分離する。

## Risks / Trade-offs

- **[256色グリッドのレンダリングパフォーマンス]** → グリッドは展開時のみレンダリングし、CSS Grid で軽量に描画。ボタン要素が多いが、仮想スクロールが必要なほどではない
- **[モバイルでの操作性]** → 小さいセルをタップしにくい可能性 → セルサイズを最低24px確保、グリッドをスクロール可能にする
- **[hex入力のバリデーション]** → 不正値で `getTextColor()` が壊れるリスク → 6桁hex以外は適用ボタンを disabled にする
